# BitPulse
BitPulse — это высокопроизводительная платформа для мониторинга криптовалют в реальном времени, построенная на базе асинхронного стека Python. Проект включает в себя систему умных уведомлений об определённых событиях в Telegram, полнотекстовый поиск по тикерам и удобное API для получения информации по тикерам.

# Стек технологий

* Backend: FastAPI (Python 3.12)
* Database: PostgreSQL (SQLAlchemy 2.0 + Alembic)
* Search Engine: Elasticsearch
* Task Queue: Celery + Redis (фоновые задачи и брокер сообщений)
* Cache & Real-time: Redis (кэширование поиска, Rate Limiting, Pub/Sub для WebSocket)
* Monitoring: Flower (Celery) + Kibana (Elasticsearch)
* DevOps: Docker, Docker Compose, GitHub Actions (CI/CD)
* Bot API: Telegram Bot (уведомления и управление алертами)

# Ключевые фичи

* **Smart Search**: Полнотекстовый поиск по тикерам с поддержкой опечаток (fuzziness) и приоритезацией (boosting) через Elasticsearch. Результаты частых запросов к поиску кэшируются в Redis (TTL-based), что снижает нагрузку на основную БД.
* **Background Processing (Celery)**: Все тяжеловесные и сетевые операции (индексация 2000+ тикеров, опрос внешних API Binance, рассылка уведомлений, расчет исторической аналитики) вынесены в асинхронные Celery-воркеры. Это гарантирует мгновенный отклик API для пользователя.
* **Real-time Alerts**: Система уведомлений в Telegram. Пользователь может управлять алертами (удалять/активировать заново) прямо из мессенджера без перехода в основной интерфейс.
* **Price History**: Сбор и хранение истории цен с интервалом в 10 минут для отрисовки графиков. Система поддерживает автоматическое удаление данных старше 24 часов
* **Rate Limiting**: Кастомный Middleware с алгоритмом Token Bucket на базе Redis для защиты API от перегрузок.
* **WebSocket Stream**: Трансляция актуальных цен тикеров на которые подписан пользователь в реальном времени.
* **Secure Auth**: JWT (Access/Refresh токены).

## Хранение данных

Система использует гибридный подход к хранению данных, выбирая наиболее подходящий инструмент для каждого типа нагрузки.

### PostgreSQL
Основная реляционная БД для хранения критически важных данных и связей:
*   **Users & Auth:** Аккаунты, хэши паролей и привязанные `tg_chat_id`.
*   **Tickers:** Справочник торговых пар.
*   **Subscriptions:** Связи Many-to-Many между пользователями и монетами.
*   **Alerts:** Конфигурации уведомлений (целевые цены, операторы сравнения).
*   **Price History:** Временные ряды цен за последние сутки (индексированные по `timestamp`).

### Redis (High-speed Storage & Cache)
Используется как брокер сообщений для Celery и быстрое хранилище(кэш):
*   `price:{symbol}` — актуальная цена тикера (обновляется раз в минуту).
*   `refresh:{username}` — Refresh-токены.
*   `search:{query}:limit:{limit}` — кэш результатов поиска в Elasticsearch.
*   `tg_secret_token:{token}` — временные коды для безопасной привязки Telegram-аккаунта.
*   `rate_limiter:{ip}` — состояние корзины токенов (алгоритм Token Bucket).

### Elasticsearch (Search Engine)
Специализированный индекс `tickers` для высокоскоростного поиска.
Гибридный маппинг полей:
`symbol` (keyword): Оптимизирован для точных совпадений торговых пар. При поиске используется коэффициент `Boosting` (^2), что поднимает точные попадания в тикер (например, BTC) выше в поисковой выдаче, чем упоминания в названиях.
`name` (text): Позволяет искать монеты по полному названию.

Реализована логика «на лету»: после получения результатов из Elasticsearch система выполняет атомарный MGET запрос к Redis, дополняя информацию о тикерах актуальными ценами прямо в поисковой выдаче.


# Быстрый запуск
## 1. Подготовка

   1. Клонируйте репозиторий:
   
   `git clone https://github.com`
   
   2. Настройте окружение:
   Создайте файл .env на основе примера `.env.example`
   
   
## 2. Запуск приложения
Проект использует Docker Profiles для управления ресурсами. Выберите подходящий режим запуска:

* Стандартный запуск (API, Worker, Beat, DB, Redis, Elastic):

    `docker compose up --build`

* Режим отладки (Дополнительно запускает Kibana и Flower):

    `docker compose --profile debug up --build`

* Режим Observability (Дополнительно запускает Prometheus и Graphana):

    `docker compose --profile metrics up --build`

## 3. Доступные сервисы

* API (Swagger): http://localhost:8000/docs
* Flower (Мониторинг задач): http://localhost:5555 (только в профиле debug)
* Kibana (Аналитика Elastic): http://localhost:5601 (только в профиле debug)

## 4. Тестирование
Для запуска тестов используется профиль `test`, который поднимает изолированную базу данных:

   1. Поднимите окружение для тестов:
   
   `docker compose --profile test up -d`
   
   2. Запустите тесты внутри контейнера приложения:
   
   `docker exec -it bitpulse_app pytest -sv`


##  Observability & Performance Testing (Наблюдаемость и Нагрузка)
В проекте реализована комплексная система мониторинга технического состояния приложения.
## Стек мониторинга:

* Prometheus: сбор метрик с FastAPI (использован prometheus-fastapi-instrumentator).
* Grafana: визуализация данных. Настроены кастомные дашборды для:
   * API: RPS, Latency (P99/P95), счетчик ошибок, Average Latency по эндпоинтам.
   * PostgreSQL: количество активных/idle соединений, нагрузка на CPU базы, размер таблиц.
   * Redis: потребление памяти кэшем, количество команд в секунду.

## Нагрузочное тестирование (Locust):
Проведено стресс-тестирование системы с помощью locust.
`locust -f locustfile.py --web-host 127.0.0.1 --web-port 8089`
Результаты оптимизации:

* До оптимизации: задержка (Latency) составляла ~200ms при 20-30 RPS из-за :
    * некорректной инициализации пула соединений SQLAlchemy (создание Engine на каждый запрос)
    * лишнему обращению к базе данных в каждом запросе
    * отсутствию индексов в базе

* После оптимизации:
* Задержка упала до 3-10ms.
   * Система стабильно держит 400+ RPS на 4 воркерах (Uvicorn).
   * Устранены ошибки TooManyConnectionsError путем настройки синглтона Engine и оптимизации max_connections в Postgres.

Что было сделано:

   1. Переход на синглтон-пул соединений в SQLAlchemy.
   2. Оптимизация JWT-авторизации (убран лишний поход в БД за объектом пользователя).
   3. Замена неэффективных .any() запросов на JOIN с индексами.
   4. Добавление индексов на столбцы `user_id` `ticker_id`
